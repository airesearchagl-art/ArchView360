🥽 VRモード（単体VR）

Meta QuestなどのWebXR対応ブラウザでは、現在表示中の360°パノラマシーンを1つだけ、HMDの向きで見回せる没入表示（VRモード）に切り替えられます。

推奨環境

推奨： Meta Quest Browser（Quest単体のブラウザ）で、HTTPS上にホストされたArchView360を開いてVRモードを使用してください。WebXRはQuest単体ブラウザでは対応しやすく、最も安定して動作します。

PC編集 → Quest確認の推奨フロー

シーンの作成・編集はPCで行い、VRでの確認はQuestで行うのが推奨の運用です。

  1. PCでArchView360を開く
  2. 画像・平面図・マーカー・比較セットを設定する
  3. 設定（プロジェクトJSON）を書き出す
  4. Quest BrowserでArchView360を開く
  5. 書き出したJSONと画像一式を読み込む
  6. VRモードで確認する

Quest Link / USB接続 / PC画面拡張について

Meta Quest LinkやUSB接続でPC画面をHMDに拡張表示する運用は、非推奨（検証扱い）です。

  • 通常の画面拡張は、ArchView360にとってのVR表示（WebXRセッション）ではありません。PC画面がHMD内に映っているだけです。
  • VRモードを使うにはWebXRの immersive-vr セッションが必要です。画面拡張だけではこのセッションは開始されません。
  • PCブラウザ＋Quest Link経由では、HMDがWebXRデバイスとして安定して認識されない場合があり、環境依存が大きいため推奨しません。

対応環境

環境VRボタン
Meta Quest Browser（Quest 2 / 3 / Pro等）有効（WebXR対応）
その他のWebXR（immersive-vr）対応ブラウザ有効
通常のPCブラウザ（Chrome/Edge/Safari等、HMD未接続）無効（自動でdisabled）
HTTP（非HTTPS）環境無効・VR開始不可
仕様： 起動時に navigator.xr.isSessionSupported('immersive-vr') でWebXR対応を判定します。非対応ブラウザではツールバーの「🥽 VRモード」ボタンが無効化され、ボタンのツールチップに「このブラウザはWebXR VRに対応していません」と表示されます。

使い方

  1. パノラマ画像、またはJSONプロジェクトを読み込みます。
  2. 通常表示でVRで見たいシーンを選択します。
  3. 上部ツールバーの「🥽 VRモード」ボタンをクリックします（WebXR対応ブラウザのみ有効）。
  4. VR表示に切り替わります。HMDの向きで360°空間を見回せます。
  5. VRを終了すると、自動的に通常表示（マウス操作）に戻ります。

Meta Questでの操作手順

  1. Quest BrowserでArchView360を開く
  2. パノラマ画像またはJSONプロジェクトを読み込む
  3. 通常表示で見たいシーンを選択する
  4. 上部ツールバーの「🥽 VRモード」を押す
  5. VR表示に入る
  6. HMDの向きで見回す
  7. 右コントローラーのAボタンで次のシーンへ、左コントローラーのXボタンで前のシーンへ切り替える（v2.15）
  8. 終了すると通常表示へ戻る

VR Cube Probe（v2.15.1〜v2.15.2で使用、v2.16.1で撤去）

撤去済み： VR開始直後に固定座標4色キューブ（Red/Blue/Green/Yellow、一辺40cm）を表示する検証専用ブロックを、v2.15.1〜v2.15.2の間、下記のHUD / Debug Panelとは完全に独立した形で使用していました。目的は「ArchView360がThree.jsシーンへ追加したMeshが、そのままVR空間内に表示されるか」を最小構成で確認することで、シーン切替・コントローラー入力はこのブロックの対象外でした。v2.16.0でVR Runtimeの移植とArchView360本体としてのQuest 3実機確認（HUD表示・Controller操作・シーン切替）が完了し、Probeの役目は終えたため、v2.16.1で撤去しました。詳しい描画経路の監査結果は VR Render Path Audit を参照してください。

VR中のシーン切替 / HUD / Debug Panel（v2.15 — WebXR Sandbox Feedback）

v2.15での方針転換： v2.13〜v2.14.1では、controller event方式 → XRSession.inputSourcesポーリング方式への刷新 → 原因切り分け用の赤/青デバッグパネル追加、と段階的にArchView360本体で試行錯誤しましたが、Meta Quest 3実機ではいずれも十分に動作しませんでした。v2.15では、別プロジェクト（WebXR-Sandbox）でQuest 3 / Quest Browser実機確認済みの方式だけを最小移植しています。

VRセッション中に、Quest Touch Plusコントローラーで前後のシーンへ切り替えられます。VRを終了せずに複数シーンを連続確認できます。

採用した方式

  • HUD：camera-forward方式を採用。 HUDメッシュをcameraの子として固定ローカルオフセットで追加します（v2.16でカメラをシーングラフに登録しcamera.add()する形に統一。v2.15時点の「毎フレームworld座標を再計算する」実装は誤移植だったため置き換え済みです。詳細はVR Runtime移植を参照）。renderer.xr.getCamera(camera)から取得したXRサブカメラへ追従・子要素化する方式（xr-camera-add）は、Sandboxでの検証の結果、期待したXRサブカメラではなく主カメラへのfallbackになっていたため不採用としました。
  • 入力：session.inputSourcesポーリングを採用。 vrLoop()内で毎フレームsession.inputSourcesを確認し、各source.gamepad.buttonsのpressed状態をfalse→trueのエッジでのみ検出します。controller event（selectstart/squeezestart）やrenderer.xr.getController()のイベントリスナーには依存しません。

VR HUD / Debug Panel

VR開始直後から、HUDパネルが必ずcameraの前方に表示されます。表示内容は次の通りです。

  • 「ArchView360 VR」「Scene N / 総数」「現在シーン名」
  • 左右Controllerの視覚ガイド（v2.18、後述）： Debug簡易/詳細のどちらでも常に表示されます
  • 簡易表示： inputSources数のみ
  • 詳細表示（Left Yで切替）： inputSources数、左右コントローラーの状態（buttons数・押下数）、最後に検出した入力、現在シーン、nav order長、リング項目数、リング表示ON/OFF状態、左右それぞれのホバー中シーン名、直近に選択したシーン名（v2.17 / v2.17.1）
  • HUDはcameraの子として、前方約2m・視線よりやや下の固定ローカルオフセットに配置されます。
  • 初期状態では自動で薄くなったり消えたりしません。常時表示で、Right Bボタン操作でのみ非表示にできます（非表示中もシーン切替操作は有効）。
  • HUD・Debug Panelの表示状態はVRセッション内だけの一時状態で、プロジェクトJSON / ZIPには保存されません。VR終了後は初期状態（表示・簡易モード）に戻ります。

📍 VR Scene Ring Navigation（v2.17 / v2.17.1）

目的： v2.16.1で完成したVR Runtimeをベースに、既存のナビゲーション対象シーンを閲覧者の周囲へリング状に等間隔表示し、レーザーポインター＋同じ手のTriggerで任意シーンへ移動できるようにする追加機能です。VR Runtimeの初期化・終了処理、camera子付けHUD、Controllerのbutton pollingロジック（Right A/Left X/Right B/Left Y）、render loopには変更を加えていません。
この機能が「していないこと」： プロジェクトデータにはシーンごとの空間座標が存在しないため、リング上の表示位置は等間隔の合成配置です。FloorMap上のピン位置をVR空間へ再現・投影する機能ではありません。
  • Scene Ring表示： VRモード開始時のみ、現在シーン以外のナビゲーション対象シーン（円形アイコン＋シーン名）を閲覧者の周囲へ等間隔に生成します。並び順は新しいJSON構造ではなく、既存のmarker.sceneId / marker.order（_getNavOrder() — FloorMapやnextScene/prevSceneと共通のナビゲーション順）です。通常表示では生成されません。シーンが1枚だけの場合は移動先が存在しないため、リングもレーザーも生成しません。
  • Billboard： THREE.Spriteとして実装しているため常にカメラ方向を向き、three.jsの標準機能（sizeAttenuation）により距離に応じて自動的に縮小表示されます。
  • Laser Pointer： 左右Quest Controllerそれぞれから細いLineを前方に表示します。renderer.xr.getController()によるthree.js標準のController姿勢トラッキングを使用し、既存のgamepad.buttons読み取り（Right A/Left X/Right B/Left Y用）とは独立しています。handedness（左右）はconnectedイベントのXRInputSourceから解決します。
  • Hover / 選択（左右独立）： hover対象は左右の手で独立に管理されます（hovered.left / hovered.right）。左手のTrigger（button[0]）は左レーザーのhover対象のみ、右手のTriggerは右レーザーのhover対象のみを選択し、反対の手のトリガーでは発火しません。handednessが取得できないinputSourceからは選択を発火しません。ホバー中の項目はハイライト表示（拡大・色変化）されます。
  • Trigger arm方式： VR開始直後は選択が無効で、各Triggerを一度離したことを確認してから選択可能（armed）になります。VRモード開始ボタンをTriggerで押したまま入っても即時選択されることはありません。選択後（Fade後）も、Triggerを一度離すまで次の選択はできません。
  • Scene切替演出： Trigger選択によるシーン切替時のみ、Fade Out（VR空間内の黒フェード、DOM側のフェードはヘッドセット内で見えないため使用しません）→ Texture Load → Fade In を行います。Fade中はリングとレーザーを非表示にし、hover・選択も無効化するため、演出中の重複選択は発生しません。テクスチャ読込の失敗・停止時も最大約5秒で必ずFade Inへ復帰し（その場合は元のシーンを維持）、黒画面が残留することはありません。Right A / Left Xによる従来の切替（nextScene() / prevScene()）の挙動は変更していません。
  • フェイルセーフ： リングの生成・毎フレーム処理で例外が発生した場合は、リングだけを破棄してVRセッション・HUD・A/X/B/Y操作を継続します。エラー内容は通常画面のVR Debug Log（last ring error行）とconsoleに表示されます。ZIP/JSON復元シーンでシーン名が空/未定義でも、Scene Nへ正規化して生成します。
  • 表示切替（v2.17.1）： 左コントローラーのMenuボタン（button[12]、実測で未使用と確認済みのボタン）でリング表示のON/OFFを切り替えられます。OFF中はリング項目・左右レーザーが非表示になり、hover・Trigger選択も完全に無効化されます（Right A/Left X/Right B/Left Yには影響しません）。ONに戻した際はTriggerのarm状態を安全にリセットするため、OFF中に押しっぱなしにしていても復帰直後に誤選択されることはありません。
  • リングの表示状態・トグル状態・ハイライト状態はVRセッション内だけの一時状態で、プロジェクトJSON / ZIPには保存されません。VR終了時に破棄され、次回VR開始時は常にON（表示）から始まります。

🧭 VR階移動ナビゲーション（v2.19）

目的： VRを終了せずに、1階・2階・3階など複数の階を移動できるようにする追加機能です。「階」＝既存のprojectState.floorplans（FloorMap平面図）で、新しいプロジェクトデータは追加していません。A/X（Right A/Left X）は引き続き現在階の中だけでのシーン移動用として維持されています。
  • 操作： 左スティック押し込み（button[3]）で1つ下の階、右スティック押し込み（button[3]）で1つ上の階へ移動します。
  • axesではなくbuttonを採用した理由： Quest Touch Plusの左右スティックのaxis番号・符号は、このプロジェクト・WebXR-Sandboxのどちらの実機記録にも存在しませんでした。番号・符号を推測で実装しないという方針のもと、実機で確定済みのスティック押し込み（button[3]）を採用しています。
  • 階の一覧・順序： projectState.floorplansの配列順（アップロード順）をそのまま使用します。文字列としての階名ソートは行いません。並べ替えUIは既存FloorMapにも存在しないため、通常画面と完全に同じ順序です。
  • 有効な階の判定： シーンが1つも紐付いていない階（マーカーが存在しない平面図）は移動対象から除外されます。
  • 現在階の決定： 既存のactiveFloorplanIdをそのまま使用します。この変数はswitchToScene()実行のたびに既存コードが自動的に現在シーンの階へ同期しているため、VR専用の新しい判定処理は追加していません。値が不正・未設定の場合のみ、現在シーンのマーカー情報から補完します。
  • 移動先シーンの決定： ①そのVRセッション内で最後に見た当該階のシーン、②未訪問なら当該階のnav order（marker.order）先頭のシーン、の優先順位で選びます。階ごとの最後に見たシーンはVRセッション内のメモリ（Map）だけに保持し、JSON/ZIPには保存しません。
  • 範囲外・循環なし： 最上階で上、最下階で下を押しても移動しません。上下端を超えて反対側の階へ循環することもありません。
  • Scene切替演出： Scene Ring選択と全く同じFade Out → switchToScene() → Fade Inの仕組みをそのまま再利用しています。階移動後はScene Ringが新しい階の内容で自動的に再生成されます。
  • 入力仕様： 左右スティックはそれぞれ独立したarmed/pressed状態を持ち、pressedの立ち上がりedgeで1回だけ発火します（長押しで連続移動しません）。VR開始直後にスティックが押されていても誤発火しません。左右が同一フレームで同時に押された場合はどちらも実行しません（離してから片方ずつ押し直すと移動できます）。Fade中・シーン切替中は新しい階移動を開始しません。
  • Ring OFF中も移動可能： 階移動はScene Ring表示のON/OFF（Left Menu）とは独立しており、Ring非表示中でも階移動できます。
  • 単一階のプロジェクト： 階が1つ（または0）のプロジェクトでは、階移動操作は実質的に無効化されます（HUDのラベルはグレーアウト表示、現在階のヘッダー表示自体も出ません）。
  • フェイルセーフ： 階移動処理は独立したtry/catchで保護されており、失敗してもVRセッション・HUD・A/X/B/Y/Trigger/Menuの既存操作を継続します。
  • Debug詳細（Left Y）： 現在階名・階index/総有効階数・現在階のシーン数・最後の階移動アクション・左右スティックのpressed/armed状態・階切替のfade状態・最後の階移動エラーを追加表示します。

🎮 VR Controller視覚ガイド（v2.18 / v2.19）

目的： VR HUDのボタン操作案内を、テキストのみの一覧から左右Controllerの簡易シルエット＋ボタン位置＋短いラベル線による視覚的なガイドへ刷新しました。既存のController入力処理（_pollVrInputSources()）・Scene Ring選択ロジック・Ring表示切替ロジックには一切変更を加えていません。HUD CanvasTexture上の描画だけを変更した見た目の改善です。図の形状・配置はプロジェクトオーナーから提供された参考図に準拠しています。
  • 左右の区別： 左Controllerは緑系の色、右Controllerは橙系の色で区別します（色分けのみで、L/R文字表記はありません）。
  • Trigger＝リング上部の輪っか： Triggerは他のボタンと同じ丸ではなく、キーホルダーのようなリング上部の小さな輪っかとして描画し、そこから直接「シーン選択」ラベルへ引き出し線を伸ばします。ボタングリッドとは独立した「最重要操作」として最上部に配置しています。
  • 2×2ボタングリッド（参考図に準拠）： リング内側に親指スティック＋3つのボタンを2×2で配置します。
    • 左：親指スティック（左上、押し込みで下の階）・Y（右上、デバッグ ON・OFF）・≡ハンバーガーアイコン（左下、リンク先 ON・OFF）・X（右下、前シーン）
    • 右：B（左上、操作方法 ON・OFF）・親指スティック（右上、押し込みで上の階）・A（左下、次シーン）・∞インフィニティアイコン（右下、メニュー／長押しで視界リセット）
    Menu（≡）・Meta（∞）は機能を直感的に示すアイコンで描画し、Mのような英字記号には頼っていません。
  • 階移動（v2.19、スティック押し込み）： 親指スティックの塗り丸自体は他のボタンと同じ位置に描かれますが、v2.19からそこにも引き出し線とラベル（下の階 / 上の階）が付きます。内部のbutton[3]という番号はHUDには表示しません。プロジェクトに階（FloorMap平面図）が1つしかない場合は、このラベルだけグレーアウト表示になり（レイアウトは変えず、無効であることだけを示す）、Ring OFF状態とは独立しています（Ring非表示中も階移動は可能です）。
  • グリップの装飾： リング下に先細りの雫形シルエットを描き、内側（もう片方のControllerに近い側）に小さなフック状の突起を添えています。実機シルエットを想起させる装飾のみで、ラベル・入力処理とは無関係です。
  • 短いラベル： 各ボタン／輪っかから短い引き出し線を伸ばし、操作名を表示します（前シーン / 次シーン / デバッグ ON・OFF / 操作方法 ON・OFF / リンク先 ON・OFF / シーン選択 / メニュー ＋ ※長押しで視界リセット / 下の階 / 上の階）。button[0]/[3]/[4]/[5]/[12]などの内部インデックスは通常HUDには大きく表示しません（本ページの「Quest Touch Plusボタン割当（実測）」表を参照してください）。
  • Ring OFF時の表現： Scene Ring表示がOFFのとき、Trigger（シーン選択）の輪っかとラベルがグレーアウトされ、選択操作が無効であることが一目で分かります。他のボタン（Y/X/Menu/B/A/Meta/階移動スティック）はRing状態に関わらず常に通常表示です。
  • 現在階の表示（v2.19）： 通常HUD上部（シーン名の下）に、階が2つ以上ある場合のみ「2階（2 / 3）」のように現在階を表示します。単一階のプロジェクトでは表示されません。
  • 右Controller Metaボタンの案内： 右Controllerの∞（Meta/Oculus）ボタンは、OS側のハードウェアボタンです。WebXRのgamepad.buttonsには公開されないため、ArchView360のコードが処理する入力ではありません。実際の挙動（短押しでシステムメニューを表示、長押しで視界をリセット）に合わせ、HUD上には常時「メニュー」「※長押しで視界リセット」という案内のみを表示し、リンク先 ON・OFF状態やTrigger操作とは独立しています（グレーアウトの対象にもなりません）。
  • 表示/非表示の連動： 操作図はHUD本体と同じCanvasTexture・同じMeshに描画されるため、Right Bで通常HUDを非表示にすると操作図も一緒に非表示になります。
  • Debugとの独立性： 操作図はDebug簡易/詳細表示の切替（Left Y）とは独立した「常時表示」領域に描画されるため、Debug表示を切り替えても操作図が消えたり壊れたりしません。
  • 更新頻度： 新しい再描画トリガーは追加していません。既存の_vrShowHud()呼び出しと、vrFrame()内の既存の間引き再描画（約20フレームに1回）にそのまま乗る形で更新されます。

🗺️ VRミニマップナビゲーション（v2.20）

目的： VRを終了せずに、現在階のFloorMapと現在位置を確認できる小さなミニマップパネルを追加する機能です。「マーカー」＝既存のprojectState.markersで、座標系も含めて通常画面のFloorMapパネルと同一のものを再利用します。新しいプロジェクトデータは追加していません。
  • 現在位置表示： 現在階に属するマーカーだけを表示します（全階を同時に表示することはありません）。現在シーンに対応するマーカーは、他マーカーより大きく、白い二重円で強調表示されます。現在シーンにマーカーが存在しない場合もVRは通常通り継続します。
  • 座標系： マーカーのx/yは、通常画面のFloorMapパネル（renderFloormapCanvas()）が使うのと全く同じ正規化座標（letterbox表示された画像領域内の[0,1]相対値）です。ミニマップは自身のCanvasサイズに対して同じletterbox-fit計算を独自に行い、この座標をそのまま再利用します。rotationOffsetはマーカーの向き表示にのみ影響する既存仕様（位置には影響しません）のため、位置表示のみの本バージョンでは使用していません。
  • FloorMap画像表示方式： HUD本体と同じ「PlaneGeometry + CanvasTexture」構成の第2の独立パネルです。個別マーカーを3Dオブジェクト化せず、Canvas上に描画し、hit-testはray-plane交点をローカル座標経由でCanvas pixel座標へ変換してキャッシュ済みマーカー座標リストと距離判定する方式のため、階が変わっても再生成されるThree.jsオブジェクトはメッシュ1個のみです。
  • FloorMap画像がない場合： 現在階に画像がない、または読み込み前の場合は、例外を出さず「FloorMapなし」というプレースホルダを表示します。
  • ray hit方式： 既存Ring用コントローラー（vrController1/2、ringHandedness）とレイをそのまま再利用します。ミニマップのマーカーはSpriteではなくMesh上のCanvas座標判定のため、Sprite特有のraycaster.camera設定は不要です（Ring選択のSprite raycastクラッシュの再発条件とは無関係）。
  • TriggerジャンプとScene Ringとの優先順位： 同じ手でScene Ring項目とミニマップマーカーの両方がhover状態になった場合、ミニマップを優先します。Trigger自体はScene Ring選択と同一のarmed/pressed状態（vrRingTrigger）を共有しており、新しい別状態を追加していないため、1回のTrigger操作で両方が同時に選択されることはありません。現在シーンのマーカーを選択した場合は不要な再遷移を行いません（no-op）。
  • Scene切替演出： 既存のFade Out → switchToScene() → Fade Inの仕組みをそのまま再利用しています。専用の遷移処理は追加していません。
  • 階移動との連携： 階移動（v2.19）でFade完了時にFloorMap画像・マーカー一覧を新しい階の内容へ自動的に再構築します。A/XやScene Ring選択など同じ階の中でのシーン切替では、現在位置マーカーの再描画のみを行い、FloorMap画像・マーカー一覧の再構築は行いません。
  • Ring OFF中も利用可能： ミニマップの表示・hover・Triggerジャンプは、Scene Ring表示のON/OFF（Left Menu）とは独立して動作します。
  • 表示切替： 未使用Controllerボタンの新規割り当てにはリスクが伴うため、本バージョンでは表示切替ボタンを追加せず、VR中は常時表示としました。表示状態はVRセッション内だけの一時的なもので、JSON/ZIPには保存されません。
  • フェイルセーフ： ミニマップの再構築・描画・raycastはそれぞれ独立したtry/catchで保護されており、失敗してもVRセッション・HUD・A/X/B/Y/Trigger/Menu/階移動の既存操作を継続します。マーカーのscene参照が不正な場合は、そのマーカーだけをスキップします。
  • リソース管理： VR開始時に1回だけThree.jsのメッシュ・Canvas・Textureを生成し、VR終了時・エラー時に確実にdisposeします。毎フレームの再生成は行いません。

通常画面のVR Debug Log

画面左下に、VRセッション実行中・終了後を通じて最後の状態が残る「VR Debug Log」を表示します。ヘッドセットを外した後でも、inputSources数・左右コントローラーのボタン数・最後の入力・next/prev/hud/debugカウント・HUDの表示状態を確認できます。ローカル表示のみで、外部送信・JSON保存・ZIP保存は一切行いません。

v2.15.2で追加： 描画経路そのものを診断するため、renderMode（normal/vr）・threeScene/cameraのUUID・renderer.xr.enabled・XRアニメーションループの稼働有無・通常rAFの稼働有無・最後のVRループ実行時刻・VRループのフレーム数を追記しました（v2.16.1でCube Probe撤去に伴いCube数の項目は削除）。詳細は VR Render Path Audit を参照してください。

コントローラー操作が効かない場合

VR HUD（簡易/詳細）・通常画面のVR Debug Logを見ることで、どこで問題が起きているか切り分けられます。

  • HUD自体が見えない場合： camera-forward方式のmesh描画自体がXR空間に乗っていない可能性があります。ページを再読み込みしてから再度VRに入ってください。
  • HUDのinputSourcesが0の場合： Quest Browser側でコントローラー自体が認識されていません。コントローラーの電源・ペアリング状態を確認し、一度いずれかのボタンを押して再アクティブ化してからお試しください。
  • Left Yで詳細表示に切り替えてもlast actionが変わらない場合： ボタンの割当（インデックス）がお使いの環境と合っていない可能性があります。詳細表示の左右buttons数を確認してください。
  • last actionは変わるがシーンが変わらない場合： 入力自体はアプリに届いているため、切替処理側の問題です。ページを再読み込みしてから再度お試しください。
  • Meta Quest Browserを最新版に更新してからお試しください（Quest 3 / Meta Quest Browserを優先対応としています）。
  • それでも切り替わらない場合は、VRセッションを終了し、ページを再読み込みしてから再度VRに入ってください。VR終了時にはブラウザのコンソールに[VR]プレフィックス付きのログ（セッション開始・入力ソース・ボタンエッジ・シーン変更・HUD切替・最終デバッグ状態）も出力されます。

通常表示に戻る方法

HMDまたはブラウザ側のVR終了操作（Questの場合は本体ボタン操作でのセッション終了等）でVRセッションを終了すると、ArchView360は自動的に通常表示に戻ります。VR中に自動回転は一時停止され、終了後は元の設定に復帰します。

対象外の機能（今回のバージョンでは非対応）

  • 比較VR（分割比較・スライダー比較のVR表示）
  • FloorMap VR（平面図のVR表示）
  • コメントVR
  • mp4パノラマ動画のVR表示
  • INSP / INSV などInsta360ネイティブ形式のVR表示
仕様： 分割比較中・スライダー比較中はVRボタンが無効化され、ツールチップに「VRモードは通常表示のみ対応です」と表示されます。通常表示に戻ると再度有効になります。VR表示中かどうかの状態はプロジェクトJSONには保存されません（一時的な閲覧状態のため）。
