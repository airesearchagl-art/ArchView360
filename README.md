> **v2.22.0 — Dirty State（未保存変更管理）基盤** — Editorで行った未保存の変更を一元的に検知し、保存前にEditor→Viewer切替・別ファイルを開く・プロジェクトのクリアといった操作で編集内容が意図せず失われることを防ぐ仕組みを追加しました。project dataを変更する関数の**成功箇所だけ**で`markProjectDirty()`を呼び、Viewerでの閲覧操作やViewer guardで拒否された操作、ダイアログのキャンセルではdirtyになりません。JSON/ZIP書き出しの成功時、および新規/空プロジェクトを開いた・読み込んだ直後は`markProjectClean()`でcleanへ戻ります。**dirtyなままEditor→Viewerへ切り替えようとした場合のみ**、共有の確認ダイアログ（「破棄して続行」／「キャンセル」）を表示します——保存操作はJSON export/ZIP exportのどちらか単一の正解を自動選択できないため、ダイアログからの直接保存は今回実装せず、キャンセルして既存の書き出しボタンから明示的に保存する2択構成にしています（詳細は後述）。Viewer→Editorの切替では確認ダイアログを出しません（Viewerは設計上dirtyになり得ないため）。非空プロジェクトへのJSON/ZIP読み込みも同じ確認ダイアログを経由しますが、非空プロジェクトへの単一画像追加（追加のみで既存データを上書きしない操作）は対象外としています。ツールバーに未保存インジケーター（●未保存）を追加し、ブラウザタブのタイトルにも`*`を付与、dirty時のみ`beforeunload`で標準警告を有効にします。`appMode`と同様、dirty状態はセッション内変数のみで保持し、プロジェクトJSON・ZIP・localStorageのいずれにも保存されません。自動保存・Undo/Redo・履歴管理・クラウド同期は今回のスコープ外です。詳細は後述の「Dirty State（未保存変更管理）基盤」を参照してください。

## 🥽 VR Mode (v2.10 — 単体VR / v2.10.1 推奨運用)

Meta QuestなどのWebXR対応ブラウザでは、上部ツールバーの「🥽 VRモード」ボタンから、現在表示中の360°パノラマシーンを単体で没入表示できます。

- **対応環境：** Meta Quest BrowserなどのWebXR（`immersive-vr`）対応ブラウザ
- **非対応環境：** 通常のPCブラウザ（WebXR未対応）ではVRボタンが自動的に無効化されます
- **対象範囲：** 静止画パノラマの**単体表示のみ**（現在選択中の1シーン）
- **対象外：** 比較モード（分割／スライダー）、FloorMap、mp4パノラマ動画、Insta360ネイティブ形式（INSP/INSV）のVR表示
- **必須要件：** HTTPS環境（`isSecureContext`）であること
- 分割比較・スライダー比較中はVRボタンが無効化され、通常表示に戻ると再度有効になります
- VRの状態（VR中かどうか）はプロジェクトJSONには保存されません（一時的な閲覧状態のため）

### 推奨運用（v2.10.1）

- **推奨：** Quest BrowserでArchView360を開き、WebXR対応ブラウザ上でVRモードを使用する
- **非推奨／検証扱い：** PCブラウザをQuest Link／USB接続／画面拡張でVR化する運用
- **理由：** WebXRはQuest単体ブラウザでは対応しやすい一方、PCブラウザ＋Quest LinkではHMDがWebXRデバイスとして安定認識されない場合があり、単なる画面拡張はWebXRの`immersive-vr`セッションにはならないため

**PC編集 → Quest確認の推奨フロー：** ①PCでArchView360を開く → ②画像・平面図・マーカー・比較セットを設定 → ③設定JSONを書き出す → ④Quest BrowserでArchView360を開く → ⑤JSONと画像一式を読み込む → ⑥VRモードで確認する

詳しくは [VRモードマニュアル](docs/vr.html) を参照してください。

### VR Scene Navigation & Debug Panel（v2.15 — WebXR Sandbox Feedback）

VRセッション中に、Quest Touch Plusコントローラーで前後のシーンへ切り替えられます。VRを終了せずに複数シーンを連続確認できるため、建築レビューでの連続ウォークスルー確認に使えます。

v2.13〜v2.14.1では、controller event方式（`selectstart`/`squeezestart`）・inputSources pollingへの刷新・原因切り分け用のデバッグパネル追加、と段階的に手を入れましたが、いずれもMeta Quest 3実機では十分に動作しませんでした。v2.15では、別プロジェクト（WebXR-Sandbox、Quest 3実機確認済み）で実証済みの方式だけをそのまま最小移植しています。

- **HUD方式：`camera-forward`を採用** — HUDメッシュを`camera`の子として固定ローカルオフセットで追加する方式（v2.16でカメラをシーングラフに登録し、`camera.add()`する形に統一。v2.15時点の「毎フレームworld座標を再計算する」実装は誤移植だったため置き換え済み。詳細は[VR Runtime移植](docs/vr-runtime-migration.md)を参照）。`renderer.xr.getCamera(camera)`から取得したXRサブカメラへの追従・アタッチ方式（`xr-camera-add`）はSandboxで検証した結果、期待したXRサブカメラではなく主カメラへのfallbackだったため不採用としました。
- **入力方式：`session.inputSources`ポーリングを採用** — `vrLoop()`内で毎フレーム`session.inputSources`を確認し、`gamepad.buttons`のpressed状態をfalse→trueのエッジでのみ検出します。controller event（`selectstart`/`squeezestart`）や`renderer.xr.getController()`のイベントリスナーには依存しません。

**Quest Touch Plusボタン割当（実測）：**

| コントローラー | ボタン | 割当 |
|---|---|---|
| 右（Right） | A（button[4]） | 次のシーンへ |
| 右（Right） | B（button[5]） | HUD表示/非表示 |
| 左（Left） | X（button[4]） | 前のシーンへ |
| 左（Left） | Y（button[5]） | Debug Panelの詳細/簡易切替 |

- グリップ（button[1]）・axes（ジョイスティック）は引き続き主要操作に使用していません。トリガー（button[0]）はVRミニマップナビゲーション（v2.20）のマーカー選択・パネル拡大に使用します（Scene Ring選択との共有armed/pressed状態は維持していますが、Scene Ring自体はv2.20で一時的に無効化されています）。左メニュー（button[12]）はv2.20からミニマップのcompact/expanded切替に使用しています（v2.17.1で追加したScene Ringリング表示切替からの用途変更、同一ボタン）。左右スティッククリック（button[3]）はVR階移動ナビゲーション（v2.19）の階移動に使用しています。
- ボタンの押下エッジ（false→true）でのみ発火し、約400msのクールダウンで連続切替を抑制します（押しっぱなしで連打にはなりません）。
- 移動順はFloorMap上の**マーカー番号（`marker.order`）順**が設定されていればその順、なければシーン一覧順です（通常表示の ← → キーと同じナビゲーション順）。
- シーン切替時はテクスチャ読み込みのため一瞬暗転する場合がありますが、VRセッションは継続します。

**VR HUD / Debug Panel：** 「ArchView360 VR」「Scene N / 総数」「現在シーン名」「現在階（v2.19、階が2つ以上ある場合のみ）」「左右Controllerの視覚ガイド（v2.18/v2.19、後述）」を常時表示し、Left Yで簡易表示（`inputSources`数のみ）と詳細表示（2カラムレイアウト、v2.20 — `inputSources`数・左右コントローラーの状態（buttons数・押下数）・最後の入力・現在シーン・nav order長・リング関連情報・階移動関連情報・ミニマップ関連情報）を切り替えられます。VRミニマップパネル（v2.20、後述）はHUDとは別のCanvasTexture/Meshで、常時表示のHUDとは独立して右下に表示されます。Controller視覚ガイドはDebug簡易/詳細のどちらでも常に表示され、Left Yでの切替による影響を受けません。HUDは`camera`の子として固定ローカルオフセット（前方約2m・視線よりやや下）に配置され、VR開始直後から必ず表示されます。Right Bでのみ非表示にできます（非表示中は視覚ガイドも含めHUD全体が非表示になり、シーン切替操作自体は有効なまま）。HUD・Debug Panelの表示状態はVRセッション内だけの一時状態で、プロジェクトJSON/ZIPには保存されません。

通常画面（VR外）の左下にも「VR Debug Log」が表示され、VRセッション終了後もヘッドセットを外した状態で入力状態を確認できます。ローカル表示のみで、外部送信・JSON保存・ZIP保存は一切行いません。

- **対応範囲：** 単体VRモードのみ。比較VR・FloorMap VR・mp4動画VR・Insta360ネイティブ形式は対象外

### 📍 VR Scene Ring Navigation (v2.17)

> **⏸️ v2.20時点で一時停止中：** Quest 3実機検証で、リング上の等間隔な合成配置と360°画像内の実際の移動方向が一致せず分かりにくいというフィードバックを受け、`VR_SCENE_RING_ENABLED = false`で**一時的に無効化**しています。無効化中はリング項目・左右レーザーが生成/表示されず、hover・Trigger選択・ray hit判定もすべて行われません（VR内の主要ナビゲーションは後述の「VRミニマップナビゲーション」が担います）。**削除ではありません** — 以下の関数・データ・ロジックはすべてそのまま温存されており、`VR_SCENE_RING_ENABLED`を`true`に戻すだけで再有効化できます（ただし、無効化と同時に用途変更した左Menuボタンの割り当てを戻す作業は別途必要です）。将来、360°画像内の実位置とリンク先の方向を対応付けたナビゲーション（本来のリング、または空間リンク）を再導入する予定です。

v2.16.1で完成したVR Runtimeをベースに、**既存のナビゲーション対象シーンを閲覧者の周囲へリング状に等間隔表示し**、Quest Controllerのレーザーポインター＋同じ手のトリガーで任意シーンへ直接移動できる機能です。既存の`nextScene()` / `prevScene()`、Controller button polling（Right A/Left X/Right B/Left Y）、camera子付けHUD、render loopには変更を加えていません。

> **注意（この機能が「していないこと」）：** プロジェクトデータにはシーンごとの空間座標が存在しないため、リング上の表示位置は等間隔の合成配置です。FloorMap上のピン位置をVR空間へ再現・投影する機能ではありません。

- **Scene Ring表示：** VRモード開始時のみ、現在シーン以外のナビゲーション対象シーン（円形アイコン＋シーン名）を閲覧者の周囲へ等間隔に生成します。並び順は新しいJSON構造ではなく、既存の`marker.sceneId` / `marker.order`（`_getNavOrder()` — FloorMapやnextScene/prevSceneと共通のナビゲーション順）です。通常表示・FloorMap・Observer Modeでは生成されません。**シーンが1枚だけの場合は移動先が存在しないため、リングもレーザーも生成しません。**
- **Billboard表示：** `THREE.Sprite`として実装しているため常にカメラ方向を向き（billboard）、three.jsの標準機能により距離に応じて自動的に縮小表示されます。
- **Laser Pointer：** 左右Quest Controllerそれぞれから細いLine（`renderer.xr.getController()`ベース）を前方に表示します。既存のgamepad.buttons読み取り（Right A/Left X/Right B/Left Y用）とは独立した、three.js標準のController姿勢トラッキングを使用しています。
- **Hover / 選択（左右独立）：** hover対象は左右の手で独立に管理されます（`hovered.left` / `hovered.right`）。左レーザーのhover対象は**左手のTrigger（button[0]）**でのみ、右レーザーのhover対象は**右手のTrigger**でのみ選択でき、反対の手のトリガーでは発火しません。handedness（左右）が取得できないinputSourceからは選択を発火しません。ホバー中の項目はハイライト表示（拡大・色変化）され、シーン名が強調されます。
- **Trigger arm方式：** VR開始直後は選択が無効で、各Triggerを**一度離したことを確認してから**選択可能（armed）になります。VRモード開始ボタンをTriggerで押したまま入っても即時選択されることはありません。選択後（Fade後）も、Triggerを一度離すまで次の選択はできません。
- **Scene切替演出：** Trigger選択によるシーン切替時のみ、Fade Out（VR空間内の黒フェード）→ Texture Load → Fade In を行います。**Fade中はリングとレーザーを非表示にし、hover・選択も無効化**するため、演出中の重複選択は発生しません。テクスチャ読込の失敗・停止時も最大約5秒で必ずFade Inへ復帰し（その場合は元のシーンを維持）、黒画面が残留することはありません。Right A / Left Xによる従来の切替（`nextScene()` / `prevScene()`）の挙動は変更していません。
- **フェイルセーフ：** リングの生成・毎フレーム処理で例外が発生した場合は、リングだけを破棄してVRセッション・HUD・A/X/B/Y操作を継続します。エラー内容は通常画面のVR Debug Logとconsoleに表示されます。ZIP/JSON復元シーンでシーン名が空/未定義でも、`Scene N`へ正規化して生成します。
- **表示切替（v2.17.1、v2.20時点で未使用）：** 左コントローラーのMenuボタン（`button[12]`、実測で未使用と確認済みのボタン）でリング表示のON/OFFを切り替える`_toggleVrRingEnabled()`/`vrRingEnabled`はそのまま残していますが、v2.20でLeft Menuの割り当て自体がミニマップのcompact/expanded切替（後述）へ変更されたため、現在この経路からは呼び出されません。ロジック自体は健全で、Left Menuの割り当てを戻すだけで再度呼び出せます。ONに戻した際はTriggerのarm状態を安全にリセットするため、OFF中に押しっぱなしにしていても復帰直後に誤選択されることはありません。状態（`vrRingEnabled`）はVRセッション内だけの一時的なもので、VR終了時に破棄され、次回VR開始時は常にON（表示）から始まります。プロジェクトJSON/ZIPには保存されません。
- **Debug：** VR Debug詳細表示に `ring items`（生成数、v2.20時点では無効化中のため常に0）・`hovered L/R`（左右それぞれのホバー中シーン名）・`selected`（直近に選択したシーン名）を追加しました。`ring enabled`（表示ON/OFF状態）の行は、`ring feature enabled`（`VR_SCENE_RING_ENABLED`フラグ自体の値）と `ring input active`（このフレームで実際にRing入力処理が動いているか）の2行へ分割しています。他のDebug項目は変更していません。

詳しくは [VRモードマニュアル](docs/vr.html) を参照してください。

### 🧭 VR階移動ナビゲーション (v2.19)

VRを終了せずに、1階・2階・3階など複数の階（既存FloorMapの平面図）を移動できる機能です。**「階」＝既存の`projectState.floorplans`（FloorMap平面図）**で、新しいプロジェクトデータは追加していません。A/X（Right A/Left X）は引き続き**現在階の中だけ**でのシーン移動用として維持されています。

- **操作：** 左スティック押し込み（`button[3]`）で1つ下の階、右スティック押し込み（`button[3]`）で1つ上の階へ移動します。実機実測で確認済みのボタンで、それまで未使用でした。
- **axesではなくbuttonを採用した理由：** Quest Touch Plusの左右スティックのaxis番号・符号は、このプロジェクト・WebXR-Sandboxのどちらの実機記録にも存在しませんでした。番号・符号を推測で実装しないという方針のもと、実機で確定済みのスティック押し込み（`button[3]`）を採用しています。
- **階の一覧・順序：** `projectState.floorplans`の配列順（アップロード順）をそのまま使用します。文字列としての階名ソートは行いません。並べ替えUIは既存FloorMapにも存在しないため、通常画面と完全に同じ順序です。
- **有効な階の判定：** シーンが1つも紐付いていない階（マーカーが存在しない平面図）は移動対象から除外されます。
- **現在階の決定：** 既存の`activeFloorplanId`をそのまま使用します。この変数は`switchToScene()`実行のたびに既存コードが自動的に現在シーンの階へ同期しているため、VR専用の新しい判定処理は追加していません。値が不正・未設定の場合のみ、現在シーンのマーカー情報から補完します。
- **移動先シーンの決定：** ①そのVRセッション内で最後に見た当該階のシーン、②未訪問なら当該階のnav order（marker.order）先頭のシーン、の優先順位で選びます。階ごとの最後に見たシーンはVRセッション内のメモリ（`Map`）だけに保持し、JSON/ZIPには保存しません。
- **範囲外・循環なし：** 最上階で上、最下階で下を押しても移動しません。上下端を超えて反対側の階へ循環することもありません。
- **Scene切替演出：** Scene Ring選択と全く同じFade Out → `switchToScene()` → Fade Inの仕組みをそのまま再利用しています。専用の遷移処理は追加していません。階移動後はScene Ringが新しい階の内容で自動的に再生成されます（Ring自体の生成・再構築ロジックも既存のまま）。
- **入力仕様：** 左右スティックはそれぞれ独立したarmed/pressed状態を持ち、`pressed`の立ち上がりedgeで1回だけ発火します（長押しで連続移動しません）。VR開始直後にスティックが押されていても誤発火しません。左右が同一フレームで同時に押された場合はどちらも実行しません（離してから片方ずつ押し直すと移動できます）。Fade中・シーン切替中は新しい階移動を開始しません。
- **Ring OFF中も移動可能：** 階移動はScene Ring表示のON/OFF（Left Menu）とは独立しており、Ring非表示中でも階移動できます。
- **単一階のプロジェクト：** 階が1つ（または0）のプロジェクトでは、階移動操作は実質的に無効化されます（HUDのラベルはグレーアウト表示、現在階のヘッダー表示自体も出ません）。
- **フェイルセーフ：** 階移動処理は独立したtry/catchで保護されており、失敗してもVRセッション・HUD・A/X/B/Y/Trigger/Menuの既存操作を継続します。

詳しくは [VRモードマニュアル](docs/vr.html) を参照してください。

### 🗺️ VRミニマップナビゲーション (v2.20)

VRを終了せずに、現在階のFloorMapと現在位置を確認できる小さなミニマップパネルです。**「マーカー」＝既存の`projectState.markers`**（座標系も含めて既存の通常画面FloorMapパネルと同一のものを再利用）で、新しいプロジェクトデータは追加していません。実機（Quest 3）検証で「常時表示の小さいパネルは位置確認には十分だが、マーカーを直接狙ってのTrigger選択が困難」「拡大中にController laserが見えず照準位置が分かりにくい」「Scene Ringの方向が実際の移動方向と一致せず分かりにくい」という3件のフィードバックを段階的に受け、**compact（通常時）／expanded（選択操作時）の2段階表示**へ改善し、**VR内の主要ナビゲーションをミニマップ主体へ**整理しました（Scene Ring一時停止の詳細は前述の「VR Scene Ring Navigation」参照）。

- **compact（通常時）：** 右下に常時表示される小さいパネルで、サイズ・位置・カメラ追従はv2.20から変更していません。現在階のFloorMapと現在位置（白い二重円）を表示し、階移動後は自動再構築されます。**compact中は個別マーカーへのレイ当たり判定を行わず、パネル全体を1つのhover/選択対象として扱います**（マーカーを直接選ぶには小さすぎるため）。パネルにレイが当たるとhoverフィードバック（枠のハイライトと「Triggerで拡大」の案内）を表示します。**compactパネルへのTriggerでシーンへ直接ジャンプすることはありません。**
- **開閉操作：** **左コントローラーのMenuボタン（`button[12]`）が、compact⇄expandedの主要な開閉操作**です（v2.17.1で追加したScene Ringリング表示切替からの用途変更、既存の同じ物理ボタンをそのまま再利用しており、新しいボタン割り当ては追加していません）。長押しでも連続トグルはせず、既存のarmed/pressed edge-detectionパターンをそのまま再利用しています。**compactパネルへのレイhover＋Triggerでも引き続きexpandedにできます**（Menuが主要操作、compact-Triggerは補助操作という位置付け）。
- **expanded（拡大操作時）：** 視界中央よりやや下に、実際にマーカーを狙って選択できるサイズで表示される独立したパネルです。compactと同じletterbox-fit計算・同じCanvas描画関数を再利用し、背景色をやや濃くして視認性を上げています。現在位置は白い二重円、マーカーはレイのhoverで拡大表示＋シーン名表示（長い名前はピクセル幅基準で省略）。**expanded中はCanvas上のマーカー座標との距離判定でhoverし**、現在シーンのマーカーを選択した場合は不要な再遷移を行いません（no-op）。
- **Controller laser表示：** **expanded表示中は左右Controllerのレーザーポインター（既存Scene Ring用の`THREE.Line`と同じ描画方式）を表示**します。始点はController、終点はray方向で、ミニマップ面に交点がある場合は小さな点（`THREE.Sprite`、常にカメラ方向を向くビルボード）で交点位置を示します。マーカーhover中は点をやや強調表示し、マーカーがない背景上でも交点位置が分かります。レーザーの終端は交点までの距離に短縮表示され（既存の`BufferGeometry`の位置属性を書き換えるだけで、毎フレームのgeometry/material再生成は行いません）、compactへ戻ると同時に既定の長さ・非表示状態にリセットされます。compact表示中はレーザー自体を表示しません（Scene Ringが有効だった頃の「Ring表示中のみレーザー表示」という設計方針を踏襲しています）。
- **拡大・縮小の仕組み：** compact/expandedで別々のジオメトリ・Canvasテクスチャは作成せず、**同一のThree.jsメッシュを`mesh.scale`で拡大縮小し、位置だけ切り替える**ことで実現しています（`worldToLocal()`はスケールも正しく打ち消すため、raycastの当たり判定計算式はcompact/expandedで共通のまま変更不要）。毎フレームのオブジェクト再生成は行いません。
- **マーカー選択 → 自動的にcompactへ復帰：** expanded中にhover中のマーカーをTriggerで選択すると既存のFade Out → `switchToScene()` → Fade Inでそのシーンへ移動し、フェード完了と同時に自動的にcompactへ戻ります。**expanded中にマーカー以外の領域（背景やパネル外）でTriggerを押した場合も、シーンジャンプせずにcompactへ戻ります**（Menuボタンでの明示的な閉じる操作と、この背景Triggerでの閉じる操作の両方が使えます）。階移動が発生した場合も同様に自動的にcompactへ戻ります。
- **誤操作防止：** Menuトグル・compact→expandedへの遷移・expanded内でのマーカー選択/クローズは、Scene Ring選択と**同一の**armed/pressed状態（`vrRingTrigger`、Menu側は専用の`vrMinimapMenuToggle`）を1箇所の判定ロジックで共有しており、1回の物理的なTrigger押下で2つのアクション（遷移＋選択）が同時に発生することはありません。compact→expanded遷移直後は、そのままTriggerを押し続けてもマーカーは選択されず、**一度Triggerを離して再度押し直すまで**選択操作を受け付けません。**左Trigger と左Menuが同一フレームで同時に押された場合**は、Menuのトグルだけが成立し、その回のTrigger押下はそのフレームでは選択/開閉に使われません（Trigger側を一度離して押し直すと通常通り動作します）。
- **Scene Ringとの関係：** `VR_SCENE_RING_ENABLED = false`によりScene Ring自体が常時無効化されているため、expanded表示中はもちろん、compact表示中も含めてScene Ringのhover・選択・ray hit判定は一切発生しません（Ring自体も現状は常に非表示です）。Scene RingのON/OFF状態（`vrRingEnabled`）はこの一連の変更によって値が変わることはありません（フラグ自体は将来の再有効化に備えてそのまま維持）。
- **hit-test半径：** expanded中のマーカー当たり判定半径は、描画上のマーカー半径よりもやや大きく設定しており（Quest 3のController照準の誤差を許容するため）、複数マーカーが近接する場合は最も近いものを優先します。個別マーカーを3Dオブジェクト化せず、Meshのraycastとcanvas座標上の距離判定のみで行います。
- **FloorMapがない場合：** 現在階にFloorMap画像がない、または画像読み込み前の場合は、例外を出さず「FloorMapなし」というプレースホルダを表示します（compact/expandedどちらでも同様）。
- **フェイルセーフ：** ミニマップの再構築・描画・raycastはそれぞれ独立したtry/catchで保護されており、失敗してもVRセッション・HUD・A/X/B/Y/Trigger/Menu/階移動の既存操作を継続します。マーカーのscene参照が不正な場合は、そのマーカーだけをスキップします。VRセッション終了時は必ずcompact状態にリセットされます。
- **リソース管理：** VR開始時に1回だけThree.jsのメッシュ・Canvas・Texture・Controller laser・laser hit-dotを生成し、VR終了時・エラー時に確実にdisposeします。毎フレームの再生成は行わず、階移動時のみFloorMap画像とマーカー一覧を再構築し、シーン切替時は現在位置マーカーの再描画のみを行い、モード切替時はメッシュのposition/scaleのみを更新します（ジオメトリ・テクスチャの再生成なし）。laser hit-dotは2つの`THREE.Sprite`で1つのCanvasTextureを共有し、VR再入場のたびに重複生成しない設計です。
- **表示状態の保存：** compact/expandedの状態はVRセッション内だけの一時的なもので、JSON/ZIPには保存されません。

詳しくは [VRモードマニュアル](docs/vr.html) を参照してください。

### 🎮 VR Controller視覚ガイド (v2.18 / v2.19 / ラベル更新 v2.20)

VR HUD内のボタン操作案内を、テキストのみの一覧から**左右Controllerの簡易シルエット＋ボタン位置＋短いラベル線**による視覚的なガイドへ刷新しました。既存のController入力処理（`_pollVrInputSources()`）には一切変更を加えていません。HUD CanvasTexture上の描画だけを変更した見た目の改善です。図の形状・配置はプロジェクトオーナーから提供された参考図に準拠しています。v2.20でLeft Menuアイコンのラベルのみ「マップ 拡大/縮小」へ更新しました（形状・色・他ボタンは無変更）。

- **左右の区別：** 左Controllerは緑系、右Controllerは橙系の色分けで区別します（色分けのみで、L/R文字表記はありません）。
- **Trigger＝リングtop部の輪っか（キーホルダー状）：** Triggerは他のボタンと同じ丸ではなく、リング上部の小さな輪っかとして表現し、そこから直接「シーン選択」ラベルへ引き出し線を伸ばします。ボタングリッドとは独立した「最重要操作」として最上部に配置しています。

## 🔀 Viewer / Editor Mode基盤 (v2.21)

ArchView360全体を、閲覧専用の**Viewer**と既存の編集機能を使える**Editor**という2つの**App Mode**に分離しました。**Panorama / Compare / Slider / VRは引き続きView Type（「何を見ているか」）**で、今回のApp Mode（「編集できるか」）とは別軸です。VRを特別扱いせず、ViewerとEditorのどちらからでも既存のVR閲覧機能（Minimapナビゲーション含む）をそのまま利用できます。VR側に新しい編集UIは追加していません。

- **初期状態：** アプリ起動時・reload後は必ずViewerから始まります。localStorage・URLパラメータからの復元は行わず、JSON/ZIPプロジェクトの読み込みでもmodeは変化しません。
- **切替UI：** ツールバー右上に現在modeのラベルと切替ボタンを表示します（Viewer時「[ Viewer ] [ Editorへ切替 ]」、Editor時「[ Editor ] [ Viewerで確認 ]」）。既存の`.header-actions`のレイアウトに沿って配置しており、モバイル・デスクトップとも大きなレイアウト変更はありません。
- **Editorへの入口の抽象化：** UIのクリックイベントから直接modeを書き換えることはせず、すべて`requestEditorAccess()`を経由します。現状は素通しの実装ですが、将来の認証・ライセンス・権限確認をこの1箇所に追加するだけで済むようにするための抽象化です。
- **「開く」と「追加」の区別：** Viewerでも単一画像・JSON・ZIPを「開いて閲覧する」ことは引き続き可能です。既存プロジェクトが空の状態での読み込みは「開く」として許可し、既にシーンが存在する状態への追加読み込みのみをEditor限定の「追加」として区別しています。
- **mutation guard：** UIを隠すだけで終わらせず、シーン追加・更新・削除・並び替え・反転、平面図追加・削除・方位補正、マーカー配置・移動・回転・削除・番号変更、グループ作成・名前変更・削除、比較セット保存・削除・名前変更、プロジェクト情報保存、JSON/ZIP書き出し・（既存プロジェクトへの）読み込みなど、project dataを変更する関数の入口に`assertEditorMode()` / `canMutateProject()`によるguardを追加しました。キーボードショートカット、右クリックのコンテキストメニュー、ドラッグ＆ドロップ、隠しDOM要素、console等からの直接呼び出し経由でもViewer中の変更は行えません。
- **非保存：** `appMode`はセッション内のJavaScript変数のみで保持し、プロジェクトJSON・ZIP・localStorageのいずれにも含まれません。既存のJSON/ZIPスキーマ・互換性は変更していません。
- **今回のスコープ外：** 未保存編集の確認ダイアログ（dirty state）の全面実装、Viewer Preview機能、認証・ライセンス・権限確認の実装、Viewer/Editorの別URL・別エントリーポイント分離は今回は行っていません（既存のdirty state実装がなかったため、今回新規に追加してもいません）。

## 💾 Dirty State（未保存変更管理）基盤 (v2.22)

Editorで行った未保存の変更を一元的に検知し、Editor→Viewer切替・別ファイルを開く・プロジェクトのクリアといった操作で編集内容が意図せず失われる、または「失われる可能性がある状態なのに警告が消える」ことを防ぎます。**自動保存機能ではありません** — 保存自体は引き続き既存の「設定を書き出し」「パッケージ書き出し」ボタンから明示的に行います。

- **一元管理：** `isProjectDirty()` / `markProjectDirty(reason)` / `markProjectClean(reason)` の3関数のみが状態を変更します。各所からの直接代入はありません。
- **dirtyになるタイミング：** project dataを変更する関数の**成功箇所だけ**でdirty化します。ファイル選択ダイアログを開いただけ・キャンセルした・Viewer guardで拒否された・同じ値を再設定しただけで実データが変わらなかった、といった操作ではdirtyになりません。
- **cleanに戻るタイミング：** JSON/ZIP書き出しの成功時、または現在のプロジェクトを実際に置き換えるOpen/Importが正常完了した直後（後述）のみです。**Editor→Viewerへ切り替えただけではcleanにしません** — 切替は保存でも破棄でもないためです。確認ダイアログで選んだだけの時点（まだ実際の読込・置換が完了する前）でもcleanにはしません。保存や読込が失敗・中断した場合は元のdirty状態を維持します。
- **保存成功の判定：** ブラウザのダウンロードAPIには「ディスクへの保存完了」を確認する手段がないため、この機能は既存の書き出し処理と同じ基準（Blob生成・ダウンロードトリガーが例外なく完了すること）を「保存成功」として扱います。
- **確認ダイアログは2つの文脈で文言・選択肢を分けています（`confirmUnsavedChanges(context)`）：**
  - **`switch-to-viewer`（Editor→Viewer切替）：** メモリ上のデータは一切削除されないため、「破棄」という言葉は使いません。文言は「保存せずViewerへ移動」、選択肢は「保存せずViewerへ移動」／「キャンセル」です。選択してもその場ではdirtyを解除しません — Viewerへ移動した後もdirty・未保存インジケーター・`beforeunload`警告はすべて維持され、Editorへ戻ってもdirtyのままです。次にOpen/Importなど実際にデータを失う操作をしようとすると、改めて確認が表示されます。
  - **`replace-project`（現在のプロジェクトを置き換えるOpen/Import・プロジェクトのクリア）：** 実際に既存データが失われ得るため、文言は「破棄して続行」のままです。選んだだけでは即座にcleanにせず、実際の読込・置換処理が成功した後にのみ`markProjectClean()`を呼びます。キャンセル時・読込失敗時は現在のプロジェクトとdirty状態を変更しません。
  - **ダイアログからの直接保存（3択目）は実装していません** — JSON書き出しとZIP書き出しという2つの独立した保存形式があり、どちらか一方をダイアログが自動選択すると意図しない形式で保存してしまうリスクがあるため、保存は既存のボタンから明示的に行う運用としています。
- **Open vs 追加/mergeの分類：** 「空のプロジェクトへ新規に読み込む」＝閲覧の延長として扱い成功時clean、「既存の非空プロジェクトへ追加・merge」＝編集として扱い成功時dirtyです（Drag&Dropも同じ判定基準）。単一画像の追加のみ、既存データを上書きしない純粋な追加操作のため確認ダイアログの対象外です（JSON/ZIP読み込みは既存マーカーのIDが衝突すると上書きされ得るため対象）。
- **`clearAllAndShowUpload()`：** WebGLコンテキストロス復旧・エラー画面からの復旧・最終シーン削除時の内部cascadeなど、内部的なUI復旧経路から呼ばれる場合はdirtyを解除しません（`{ markClean: false }`、デフォルト）。明示的な「プロジェクトをクリア」操作から確認ダイアログを経て呼ばれる場合のみ`{ markClean: true }`でdirtyを解除します。
- **UI表示：** ツールバー右上（App Mode切替の隣）に未保存インジケーター「● 未保存」を表示し、ブラウザタブのタイトルにも`*`を付与します。Viewerへ切り替えている間もdirtyであれば表示されたままで、cleanに戻ると両方とも確実に解除されます。
- **beforeunload：** `beforeunload`リスナーは常時登録済みですが、実際に警告を出すのはdirty時のみです。動的な登録・解除は行わず、内部でdirty状態を都度確認する方式のため、状態とのズレが起きません。Viewerへ切り替えているだけでは警告は抑制されず、dirtyな編集内容がある限りViewer閲覧中も維持されます。
- **非保存：** `appMode`と同じく、dirty状態はセッション内のJavaScript変数のみで保持し、プロジェクトJSON・ZIP・localStorageのいずれにも含まれません。reload後は常にcleanかつViewerから始まります。
- **今回のスコープ外：** 自動保存、localStorageへの編集途中データ保存、Undo/Redo、履歴管理、クラウド同期、ダイアログからの直接保存（3択目）は今回のスコープ外です。

## 📦 Project Package Edition (v2.12)

プロジェクト全体（設定情報＋パノラマ画像＋平面図画像）を1つのZIPファイルとして書き出し・読み込みできます。社内展開・施主への共有が、ファイルを個別に管理する必要なく1ファイルで完結します。

### ZIPインポート

- 初期画面の「プロジェクトZIPを開く」ボタン、またはドロップゾーンへの `.zip` ドラッグ＆ドロップで読み込み開始
- ビューア画面が表示されている状態でも `.zip` をドラッグ＆ドロップして読み込み可能
- ZIP内から `project.json` を自動検出（ルート直下／サブフォルダ内のいずれでも対応）
- `panoramas/` ・ `floorplans/` フォルダ内の画像を `fileName` で自動照合・復元
- シーン・FloorMap・マーカー・グループ・比較セット・プロジェクト情報をすべて復元
- ZIP内に対応する画像が無い場合はその分だけ復元せず、不足ファイル名をトースト通知（可能な範囲で復元）

### ZIPエクスポート

- 上部ツールバー「📦 パッケージ書き出し」から実行
- 現在の `project.json` を生成し、読み込み済みのパノラマ画像を `panoramas/`、平面図画像を `floorplans/` に格納
- `README.txt` を同梱
- ファイル名：`ArchView360_Project_YYYYMMDD_HHMM.zip`

### 既存の設定JSONとの使い分け

| | 設定JSON（既存） | プロジェクトZIP（v2.12新規） |
|---|---|---|
| 内容 | 設定情報のみ（軽量） | 設定情報＋画像一式 |
| 用途 | 画像を別管理／差分更新 | 社内展開・施主への一括共有・PC間の引っ越し |
| 書き出し | 「JSON書き出し」ボタン | 「📦 パッケージ書き出し」ボタン |
| 読み込み | 「JSON読み込み」ボタン | 「📦 パッケージ読み込み」ボタン |

既存の「設定JSONを書き出し／読み込み」機能はそのまま残しています。

### JSZipの利用とライセンス

ZIPの生成・展開には [JSZip](https://stuk.github.io/jszip/) 3.10.1 を使用しています。CDNには依存せず、`vendor/jszip.min.js` としてローカルに同梱しているため、**オフラインでも動作します**。JSZipは MIT License または GPLv3 のデュアルライセンスで提供されています（`(MIT OR GPL-3.0-or-later)`）。

詳しくは [プロジェクトJSON / プロジェクトZIP マニュアル](docs/project-json.html) を参照してください。

## 🔒 Security Policy

ArchView360は**ローカル完結型**のツールです。サーバー側のバックエンドを持たず、以下のデータは一切外部送信されません。

- パノラマ画像
- 平面図（FloorMap）画像
- プロジェクトJSON
- プロジェクトZIP（v2.12、JSZipによりブラウザ内で展開・生成）

すべての処理はブラウザ内（クライアントサイド）で完結します。読み込んだファイルを外部に送信するAPIエンドポイントは存在しません。プロジェクトの共有が必要な場合は、JSON＋画像ファイルをユーザー自身が任意の方法で送付してください。詳細は [セキュリティポリシー](docs/security.html) を参照してください。

## サムネイル付きシーン一覧

アップロードした各パノラマ画像のサムネイルを自動生成してサイドバーに表示します。

- **自動生成**: ブラウザ内の Canvas API で 240×135px のプレビュー画像を生成（サーバー送信なし）
- **ホバー拡大**: サムネイルにマウスを乗せると 240×135px の拡大プレビューをサイドバー横に表示
- **アクティブ強調**: 現在表示中のシーンは青い枠で強調
- **クリック切替**: サムネイルをクリックするとそのシーンに切替

## ドラッグ＆ドロップ並び替え

サイドバーのシーンをドラッグして並び替えることができます。

```
使い方:
1. 並び替えたいシーンをドラッグ
2. 挿入したい位置に移動（挿入線が表示される）
3. ドロップで並び替え完了

並び替え後:
- シーン番号が自動更新される
- 現在表示中のシーンはそのまま維持される
- 比較A/Bに選択されているシーンも追跡される
- ← → キーの移動順も並び替え後の順序に従う
```

### 建築用途の並び替え例

```
並び替え前（アップロード順）:
  1. 3F_執務室.jpg
  2. 1F_エントランス.jpg
  3. 外観_東.jpg
  4. 2F_食堂.jpg

プレゼン順に並び替え後:
  1. 外観_東.jpg      ← 外から確認
  2. 1F_エントランス  ← 入口の雰囲気
  3. 2F_食堂          ← 共用スペース
  4. 3F_執務室        ← 主要執務スペース
```

## 比較モードの使い方

### 分割比較

```
1. 2つ以上のシーンを読み込む
2. 上部ツールバー「⊞ 分割比較」をクリック（またはCキー）
3. A・Bのドロップダウンで比較対象を選択
4. ドラッグで視点移動（視点同期ONのとき両面が連動）
5. 「← 通常表示」ボタンまたはEscで通常表示へ戻る
```

### スライダー比較

```
1. 2つ以上のシーンを読み込む
2. 上部ツールバー「◫ スライダー比較」をクリック（またはSキー）
3. 中央のスライダーをドラッグして境界を移動
4. 左側がA、右側がB（中央ハンドルで境界を調整）
```

### 視点同期 ON/OFF (Lキー)

```
視点同期 ON（デフォルト）:
  - AをドラッグするとBも同じ視点で動く
  - 同じアングルで仕上げ違いを比較できる

視点同期 OFF:
  - AとBを別々のアングルで操作できる
  - 「部屋の奥から見た案A」と「廊下から見た案B」を同時確認
  - 異なる視点の確認に使える
```

### 建築用途の比較例

| ユースケース | シーン A | シーン B |
|-------------|---------|---------|
| 壁あり / なし比較 | 壁あり案 | 壁なし案 |
| 仕上げ違い比較 | タイル仕上げ | 塗装仕上げ |
| 昼景 / 夕景 | 昼間レンダリング | 夕景レンダリング |
| 提案 A / B | 提案Aリビング | 提案Bリビング |
| 同視点での確認 | 視点同期ON | 同一アングル比較 |
| 別方向の確認 | 視点同期OFF | Aはエントランス、Bは廊下 |

## 🧪 自動テスト

既存機能を壊さずに再実行できる最小の自動回帰テスト基盤です。手動確認を置き換えるものではなく、代表的な操作の回帰を検知するための最小限のsmoke testです。

**必要環境:** Node.js 18以上（`@playwright/test`自体の要件）

**セットアップ:**

```bash
npm install
npm run playwright:install   # 初回のみ、Chromiumをローカルへ取得
```

**実行:**

```bash
npm test              # 全件をheadlessで実行
npm run test:headed   # ブラウザを表示して実行
npm run test:debug    # Playwright Inspectorでデバッグ
```

**対象範囲:** 初期表示、Viewer/Editorモード切替、Viewerでのmutation guard（隠しDOM要素経由の回帰確認）、Dirty Stateの基本遷移、JSON保存後のclean化、ローカルテスト用静的サーバー（`tests/server.js`）のパストラバーサル対策。テストは`tests/fixtures/`内の自作の最小画像のみを使用し、実案件データは一切参照しません。

**CI:** 現時点ではGitHub Actions等のCIは未追加です（詳細は該当PR参照）。ローカルでの`npm test`実行が現在の唯一の自動検証手段です。

## 注意点

- 非常に高解像度（8K以上）の画像はブラウザのメモリを大量に消費します
- iOS Safari では WebP の対応が限定的な場合があります。JPEG を推奨します
- `file://` プロトコルより `npx serve .` などのローカルサーバー経由を推奨します
- VRモード（v2.10）はHTTPS環境（`isSecureContext`）が必要です。WebXR非対応ブラウザではVRボタンが自動的に無効化されます
- VRモードは静止画パノラマの単体表示のみ対応です。比較モード・mp4パノラマ動画・Insta360ネイティブ形式（INSP/INSV）のVR表示は対象外です

## FloorMap Navigator の使い方

```
1. ツールバー「🗺 平面図」→ 平面図画像をアップロード
2. 画面右下に FloorMap Navigator が表示される
3. 「📍」ボタンでマーカー配置モードON
4. 平面図上をクリック → 現在表示中のシーンのマーカーを配置（現在の視点方向を自動取得）
5. マーカークリック → そのシーンへ移動（配置モード・通常モード両方で動作）
6. マーカードラッグ → 位置を移動
7. マーカー選択後、情報パネルに名称・シーン・向きが表示される
8. ✏ボタンまたはダブルクリック → マーカー名称をインライン編集
9. ↺/↻ で15°刻みで向きを調整
10. 視野角コーン（扇形）がアクティブマーカーの向きとFOVをリアルタイムで表示
11. マーカーにホバーするとツールチップで名称とシーン名を表示
12. パネル下部のマーカー一覧でシーン移動・現在位置ハイライト
13. パネル上端をドラッグして高さを自由調整、または S/M/L/XL ボタンで幅プリセット切替
14. − で折りたたみ
15. 複数平面図を追加すると切替ドロップダウンが表示される
16. 360°ビューをドラッグ中はマーカーの矢印がリアルタイムで更新される
```

## マーカーの視野角コーン仕様

- アクティブ（現在シーン）のマーカーのみ扇形コーンを表示
- コーン角度 = 現在のFOV（ホイールズームに連動）
- コーン長さ = 55px（固定）
- 色 = 半透明ブルー（fill: rgba(79,124,255,0.15)）
- リアルタイム更新: マウスドラッグ・ホイールズーム・リセット・シーン切替すべてに追従
- 左右反転シーンも対応（thetaToFloorRotation の符号で吸収）
- コーンはJSONに保存されない（表示専用）

## JSON Export / Import の使い方

### 推奨フォルダ構成

```
Project_360/
├─ archview360-project.json
├─ floorplans/
│   ├─ 1F.jpg
│   └─ 2F.jpg
└─ panoramas/
    ├─ A-1.jpg
    ├─ A-2.jpg
    └─ Entrance.jpg
```

### Export

```
ツールバー「⬇ 書き出し」→ archview360-project.json をダウンロード
※ JSONには画像データが含まれません（ファイル名のみ）
※ 画像は別途フォルダで管理してください
```

### Import（別PCで復元する場合）

```
1. ツールバー「⬆ 読み込み」→ archview360-project.json を選択
2. 必要なファイル一覧が表示される
3. 「画像ファイルを選択して復元」→ 平面図＋パノラマ画像を一括選択
4. fileName が完全一致したファイルのみ復元される
```

> ⚠️ 別PCで復元するには画像ファイル一式が必要です。JSONだけでは画像を復元できません。

### JSON スキーマ（v2.2）

```json
{
  "appVersion": "2.0",
  "exportedAt":  "...",
  "projectName": "...",
  "scenes":      [{ "id", "name", "fileName", "flipped" }],
  "floorplans":  [{ "id", "name", "fileName" }],
  "markers":     [{ "id", "floorplanId", "sceneId", "x", "y", "rotation" }],
  "compareSets": [...]
}
```

`x` / `y` は画像内の比率（0〜1）で保存。`imageUrl` フィールドは将来の ArchShare360 対応のため予約済み。

## 比較セット保存の使い方

```
1. 分割比較またはスライダー比較モードに入る
2. A・Bシーンを選択、レイアウト・スライダー位置を調整
3. ツールバー「💾 セット保存」をクリック
4. アプリ内モーダルでセット名を入力して「保存」
5. 左サイドバーの「比較セット」欄にカード表示される
6. カードをクリックまたは「開く」ボタンで比較設定を復元
7. ✏ で名前変更（モーダル）、× で削除
```

### 保存場所と制約

| 項目 | 内容 |
|------|------|
| 保存場所 | このブラウザの localStorage |
| 画像データ | **保存されません**（シーン名・設定のみ） |
| 共有 | **別PCや別ブラウザには共有されません** |
| 削除 | セットが参照するシーンを削除した場合は復元時にエラー通知 |

## 初期画面から設定JSONを開く

### 方法A：ボタンから開く

```
1. 起動画面の「設定JSONを開く」ボタンをクリック
2. archview360-project.json を選択
3. 必要ファイル一覧（パノラマ・平面図）が表示される
4. 「画像ファイルを選択して復元」で画像ファイルを一括選択
5. fileName が完全一致したものだけ復元される
6. トーストにシーン/平面図/マーカー/比較セット数が表示される
```

### 方法B：JSONをドロップゾーンにドラッグ＆ドロップ

```
ケース1 — JSONのみドロップ:
  1. .json ファイルをドロップゾーンにドラッグ＆ドロップ
  2. 復元モーダルが表示される
  3. 「画像ファイルを選択して復元」で画像を選択

ケース2 — JSON + 画像を同時ドロップ:
  1. .json と画像ファイルをまとめてドロップゾーンにドラッグ＆ドロップ
  2. JSON内の fileName と画像ファイル名を自動照合
  3. 一致する画像を使ってそのまま復元される（モーダル省略可能）
  4. 不足ファイルがある場合はその分だけ復元されない

ケース3 — 画像のみドロップ:
  → 従来通り新規プロジェクトとして開始
```

> ⚠️ 別PCで復元するには画像ファイル一式が必要です。JSONだけでは画像を復元できません。**ファイル名は完全一致**が必要です（大文字小文字も含む）。

## FloorMap ピン方位仕様

- パノラマビューを右方向に回すと、FloorMapのピン矢印も右方向に回転します
- パノラマビューを左方向に回すと、ピン矢印も左方向に回転します
- 左右反転（flipH=true）シーンでも同じ方向感で動作します
- ピン配置時（クリック）も現在の視線方向を自動取得して矢印方向を設定します
- ↺/↻ ボタンで15°単位の手動調整が可能です
